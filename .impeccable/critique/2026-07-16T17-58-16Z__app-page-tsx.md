## Overall Impression

The tool has a strong identity — dark terminal aesthetic, double-bezel cards, ForgeHero interaction — and genuinely good accessibility foundations (reduced-motion, focus rings, `aria-live` regions, skip-to-content). The color system is cohesive and semantic. The architecture is well-factored.

But it carries developer-tool blind spots: no keyboard shortcuts (score 1/4), sluggish 700ms animations, no guardrails before writing PRs to real repos, and a placeholder text that fails WCAG AA in an accessibility tool. The eyebrow-label scaffolding makes the UI feel templated rather than crafted.

**Biggest opportunity**: Fix the placeholder contrast (P0 — embarrassing for a11y tool), add keyboard shortcuts, halve animation durations, and add a PR confirmation step. These are the gap between "functional prototype" and "shippable developer tool."

## What's Working

1. **Color system and dark theme**: Ink background (#1a1717) with semantic amber/fail/focus is genre-appropriate and avoids the generic "SaaS cream" palette. Surface opacity layering creates hardware-like depth. Not generic.

2. **ForgeHero terminal**: Interactive terminal that accepts HTML input and renders diff output. Shows what the product does instead of telling. The best-designed surface in the app.

3. **Accessibility foundations**: `prefers-reduced-motion` wired everywhere. Focus-visible rings. `aria-live="polite"` on live regions. Skip-to-content link. Semantic heading structure. For a tool that sells accessibility, this consistency earns credibility.

## Priority Issues

### P0 — Placeholder text contrast fails WCAG AA SC 1.4.3

- **What**: `page.tsx:155` text input uses `placeholder:opacity-40` on `--color-muted` (#9a9897), computing to ~#4d4a4a. Contrast ratio **2.8:1** against #1a1717 bg.
- **Why it matters**: An accessibility tool with illegible placeholder text is a credibility-destroying contradiction. Worst possible first impression for screen reader users or low-vision developers.
- **Fix**: Remove `placeholder:opacity-40`. Use `--color-muted` at full opacity (5.86:1, passes AA). Or use a lighter muted token for placeholders.
- **Fix location**: `page.tsx:155` — input placeholder class. `ForgeHero.tsx` textarea if same pattern applies.
- **Suggested command**: `/impeccable audit`

### P0 — No guardrails before creating PRs on real repos

- **What**: `POST /api/pr` fires for each fix group without user confirmation. The `dryRun` flag exists but is surfaced as small text in PRCard, not a pre-flight choice.
- **Why it matters**: One bad regex replace or hallucinated fix can break production. The design has zero friction before the most destructive action in the workflow.
- **Fix**: Add confirmation overlay before "Fix & PR" phase showing: (a) repo URL, (b) PR count, (c) explicit dry-run toggle, (d) "Confirm & create PRs" vs "Cancel" button.
- **Suggested command**: `/impeccable shape` for the confirmation flow, then build

### P0 — 700ms transitions make UI feel sluggish

- **What**: Every component uses `transition-all duration-700 ease-[cubic-bezier(0.32,0.72,0,1)]`. Nav, buttons, tabs, overlay — all 700ms.
- **Why it matters**: Product register says 150-250ms for tool UI. 700ms is slower than every OS animation. A developer tool should feel snappy, not syrupy. The premium curve is good — halve the duration.
- **Fix**: Replace `duration-700` → `duration-250`. Reserve 500-700ms only for hero reveal entry.
- **Suggested command**: `/impeccable polish`

### P1 — Overused eyebrow label pattern (main AI tell)

- **What**: `font-mono text-[10px] uppercase tracking-wider` appears ~10× across sections: hero subtitle, dashboard, score card, issues, downloads, before/after labels, fix preview, PR section.
- **Why it matters**: This is the #1 AI-generated layout hallmark. Fragments content into labeled boxes instead of letting it flow. Makes every section feel like a template slot.
- **Fix**: Keep only the hero subtitle ("make your website accessible to everyone"). Remove from dashboard empty state cards (score/issues/downloads are self-evident). Remove "fix preview" label — diffs are self-describing. Keep section headers as regular sentence-case text.
- **Suggested command**: `/impeccable quieter`

### P1 — page.tsx is a 388-line monolith

- **What**: State definitions, event handlers, 4 view conditions, and all rendering in one component. Dashboard alone ~200 inline lines.
- **Why it matters**: Every change risks side effects. No component reuse. Testing requires full page mount. Debugging requires reading 388 lines.
- **Fix**: Extract Dashboard into `src/components/Dashboard.tsx`. Move scan form into `src/components/ScanForm.tsx`. Keep page.tsx as thin orchestration.
- **Suggested command**: `/impeccable distill`

### P2 — No keyboard shortcuts (developer tool)

- **What**: Zero accelerators. Every action requires pointer interaction.
- **Why it matters**: Target audience is developers. Developer tools without keyboard shortcuts feel broken. Immediately noticed by power users.
- **Fix**: Add `s` → focus scan input, `Enter` → submit, `c` → cancel, `1/2/3` → tab switching.
- **Suggested command**: `/impeccable delight`

### P2 — Empty state shows fabricated data

- **What**: Dashboard empty state hardcodes "12 missing-alt-text, 8 low-contrast..." with specific counts.
- **Why it matters**: Sets false expectations. User scans a clean repo and wonders what's wrong. The preview is a best-case scenario, not realistic.
- **Fix**: Show generic dashes and disabled buttons. Remove fabricated violation list.
- **Suggested command**: `/impeccable onboard`

### P3 — Button loading state uses "..." text

- **What**: During scan, submit button shows `"..."` text. No visual progress indicator on the button itself.
- **Why it matters**: "..." is the most generic loading indicator. Doesn't communicate work-in-progress. Missing `aria-label` update.
- **Fix**: Add `loading` prop to Button, show inline CSS spinner. Update `aria-label` to "Scanning...".
- **Suggested command**: `/impeccable harden`

### P3 — ForgeHero terminal shows rendering artifact

- **What**: Terminal diff output shows `<img src="/hero.png" / alt="descriptive image">` with a misplaced `/` before `alt` attribute.
- **Why it matters**: The simulation has a bug. For a tool that sells fix quality, visible bugs in the hero demo undermine trust.
- **Fix**: Correct the template string in `ForgeHero.tsx` that generates the diff output.
- **Suggested command**: `/impeccable audit`

## Persona Red Flags

**Selected personas**: Alex (Power User), Sam (Accessibility-Dependent), Jordan (First-Timer)

### Alex (Power User)

- **No keyboard shortcuts** for any action. Every scan submission, navigation, download requires pointer. Alex abandons after second click to download report.
- **No batch operations**. Violations shown as paginated-like grid (though all visible). No filter, sort, or group-by options.
- **What breaks**: Alex tries `Ctrl+Enter` to submit scan URL. Nothing happens. Looks for `Esc` to close mobile menu overlay — the div[role=dialog] pattern doesn't trap or dismiss via Escape natively.
- **Risk**: High abandonment. Devs expect keyboard affinity from CLI-adjacent tools.

### Sam (Accessibility-Dependent)

- **Placeholder text fails WCAG AA** at 2.8:1 contrast. Screen reader will still read it, but low-vision users who don't use screen readers cannot read what's in the field before typing.
- **"..." loading button** — `aria-label` not updated to "Scanning..." or "Cancel scan". Screen reader announces generic "button" or "..." which is non-semantic.
- **Tab key navigation**: Skip-to-content link uses `-left-[9999px]` pattern rather than `sr-only focus:not-sr-only`. Works functionally but less tested pattern.
- **No light mode**: Dark-only theme means users with light-sensitive conditions or those who need high-contrast in bright ambient light have no alternative.
- **What works**: Skip-to-content exists. Focus-visible rings are consistently styled. `aria-live="polite"` on activity log. Stepper `aria-current="step"`. `aria-label` on hamburger button.

### Jordan (First-Timer)

- **"Guide" and "Docs" nav labels**: Generic. Jordan doesn't know what "Guide" contains vs "Docs." From the label alone, they're indistinguishable.
- **Dashboard empty state**: Shows fabricated data (12, 8, 6 counts). Jordan thinks a scan already happened. May click "download report" and get confused when nothing happens (the download buttons have no disabled state).
- **No severity explanation**: Violation cards show "critical"/"serious"/"moderate" badges but nowhere explains what those mean. Jordan has to guess which issues matter.
- **What works**: The hero explains the product in one sentence. The input has a clear `$` prompt and placeholder URL example. ForgeHero interactive demo teaches through doing.

## Minor Observations

- Nav overlay uses `invisible` + `opacity-0` — redundant (opacity-0 already hides element)
- Stepper `min-w-[140px]` may cause horizontal scroll on <320px viewports
- Footer `text-[10px]` borderline for WCAG SC 1.4.4 (text resizable to 200%)
- No `prefers-color-scheme` light mode detection — dark-only
- `forge-breathe` radial gradient uses `transparent 70%` — `rgba(232,168,56,0)` would be smoother
- ScoreCard has no loading state — data could flash empty then fill
- Mobile overlay uses `div[role=dialog]` not native `<dialog>` — misses built-in focus trapping and Escape handling
- Font preloads in layout.tsx cause 3 console warnings (unused until React hydrates)

## Questions to Consider

1. **An a11y tool with placeholder text that fails WCAG AA** — is this the right hill to die on? It's a two-character fix (remove `opacity-40`) and the single biggest credibility risk.
2. **Dark-only theme** — 1/6 developers use light mode for reading code in bright environments. Is the terminal identity worth excluding them?
3. **Three typefaces** — the product register says one family is often right. Does the typography budget serve the user or the designer's desire for variety?
4. **ForgeHero is the best surface in the app** — why is it only visible on the hero view? Could the terminal appear inline in results, letting users edit proposed fixes before PR creation?
5. **Zero keyboard shortcuts** — do you expect your developer users to click "Download Report" every time, or is there an unshipped accelerator layer?
